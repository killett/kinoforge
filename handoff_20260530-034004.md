## 4. Splitter — architectural decisions (locked)

### 4.1 Why an ABC + registry, not a single function

Considered three scopes during brainstorm:
- Heuristic-only pure function — minimum surface, but limits LLM/scene-detect plug-ins.
- **Pluggable ABC + heuristic default (chosen).** Future strategies (LLM-semantic, scene-detect) slot in as adapters under `src/kinoforge/splitters/<name>/` without touching core.
- Heuristic + LLM in same PR — too much scope; would force the engine-kind ADR now.

The ABC matches the four pre-existing axes exactly. Pattern is identical: `register_splitter("name", factory)` + `get_splitter(name)` + `UnknownAdapter` on miss with the key in the message.

### 4.2 The contract

```python
class Splitter(ABC):
    name: str

    @abstractmethod
    def split(
        self, prompt: str, profile: ModelProfile, params: dict
    ) -> list[Segment]: ...
```

**Postconditions** the orchestrator relies on:
1. `len(result) >= 1`.
2. Every `Segment.prompt` is non-empty after stripping.
3. `Segment.assets == []` and `Segment.params == {}` for every returned segment (orchestrator owns asset attachment; `strategy.decide` owns param merging).
4. Order preserved — input narrative order maps to output index order.

### 4.3 `HeuristicSplitter` — paragraph-per-segment on blank-line markers

```python
_PARAGRAPH_BREAK = re.compile(r"\n\s*\n+")
```

- Marker: blank line (regex requires at least two newlines with optional whitespace between).
- Each non-empty paragraph (after strip) → one `Segment`.
- Runs of newlines collapse — no empty middle segments.
- Single-paragraph passthrough preserves the existing single-segment behaviour exactly.
- All-whitespace prompt → `ValueError` (defensive; `validate_request` upstream should already reject empty prompts).
- `profile` and `params` accepted for ABC compliance but **not consulted** by the heuristic.
- Caller's `params` dict is never mutated (explicit test: T-12).

Marker choice rationale: blank lines play nicely with YAML multi-line strings and editors, require zero new syntax for users, and are unambiguous. Considered alternatives — `---`, both, and `[scene]…[/scene]` tagged blocks — explicitly deferred.

### 4.4 Where the splitter sits in `orchestrator.generate()`

Step 6 (was a DEFERRED stub at `orchestrator.py:404`) is now:

```python
validated = validate_request(profile, request, accepted_kinds=accepted_kinds)

splitter = registry.get_splitter(cfg.splitter.kind)()
prompt_segments = splitter.split(validated.prompt, profile, {})

if prompt_segments and validated.assets:
    prompt_segments[0] = dataclasses.replace(
        prompt_segments[0], assets=list(validated.assets)
    )
```

Then at the bottom:

```python
artifact = stage.run(request, segments_override=prompt_segments)
```

**Assets attach to segment 0 only.** Continuity (issue #02) will fill segments 1..N-1 with previous-frame conditioning when implemented; until then, downstream segments render condition-free.

### 4.5 `validate_request` discipline — exactly once per `generate()`

Pre-splitter: orchestrator called `validate_request` at step 5 (discarded the return) AND `GenerateClipStage.run` re-called it internally → 2x per generate.

The integration commit (`d1828b7`) consolidated this. Orchestrator now captures `validated = validate_request(...)`. `GenerateClipStage.run` was reordered so the override branch comes first and `validate_request` is only called in the no-override branch:

```python
if segments_override is not None:
    segments = segments_override
else:
    validated = validate_request(
        self.profile, request, accepted_kinds=self.accepted_kinds
    )
    segments = [Segment(prompt=validated.prompt, assets=list(validated.assets))]
```

In the production path the orchestrator always passes `segments_override`, so validation runs exactly once (in the orchestrator). The single-segment direct-stage path (used by tests that pre-build segments) still validates inside the stage. Docstring on `GenerateClipStage.run` was tightened to make the conditional explicit (commit `999e39d`).

### 4.6 Config

```python
class SplitterConfig(BaseModel):
    kind: str = "heuristic"

class Config(BaseModel):
    ...
    splitter: SplitterConfig = Field(default_factory=SplitterConfig)
```

- **No validator on `kind`.** Unknown kinds parse at load time. `UnknownAdapter` surfaces at `generate()` via `registry.get_splitter` — matches the existing engine/provider behaviour.
- `default_factory=SplitterConfig` (NOT a shared instance default) — avoids any accidental shared mutable state if the model is reused.
- All four example configs (`local-fake`, `diffusers`, `wan`, `hosted`) still parse unchanged; covered by `tests/test_examples.py`.

### 4.7 Registration trigger

`HeuristicSplitter` lives in `core/splitter.py`. To guarantee its `register_splitter("heuristic", ...)` line fires before any `get_splitter("heuristic")` call, `src/kinoforge/core/__init__.py` (previously empty) gained:

```python
# Trigger in-core self-registrations (HeuristicSplitter).
from kinoforge.core import splitter  # noqa: F401
```

This keeps the §3.1 architectural invariant clean: `_adapters.py` is still the **only** place outside tests that imports concrete adapter packages (providers/engines/sources/stores). In-core defaults wire themselves up via `core/__init__.py`. Future splitter adapters (LLM-semantic, scene-detect) will live under `src/kinoforge/splitters/<name>/` and register via `_adapters.py` like the other axis adapters — at which point `tests/test_core_invariant.py` will need to extend its allowlist for the new `splitters/` directory in the same shape it already does for `providers/`, `engines/`, `sources/`, `stores/`.

### 4.8 What the splitter does NOT do (explicit deferrals)

| Item | Where it lives later |
|---|---|
| LLM-semantic splitter | Adapter under `src/kinoforge/splitters/llm/`. |
| Scene-detect splitter | Same pattern. |
| Per-segment `params` overrides (seed offset, cfg drift) | Future Splitter subclass; ABC supports it without contract change. |
| Per-segment metadata syntax (e.g. `[scene seed=42]`) | Deferred until needed. |
| Continuity (tail-frame → next segment's `init_image`) | Tracking-issue draft #02. Splitter contract leaves segments 1..N-1 `assets` empty for #02 to fill. |
| Stitching across N artifacts (concat / crossfade) | Separate concern; `GenerateClipStage.run` still returns `results[-1]`. |
| Duration-budget enforcement / estimation | User owns segment authoring via marker placement. Future strategies may enforce. |

## 5. Patterns established (use these for future work)

Carries forward from the prior handoff §4 — still all in force:
- **Injected I/O seams** on every adapter (constructor params with stdlib defaults; tests pass spies; production uses defaults). No real network/subprocess/git/GPU in tests; loopback fixture only for the downloader.
- **Self-registration on import** — engines/providers/stores register as zero-arg factories; sources register as instances (dispatch by `handles(ref)`).
- **Source dispatch by behaviour, not key** — first source whose `handles(ref)` returns True wins.
- **Stage protocol + pool-swap** — `GenerateClipStage` talks only to `BackendPool` + `ArtifactStore` + `ModelProfile`. `SequentialPool` is the default; future `ConcurrentPool` is a drop-in.
- **Strategy as a pure function** — `core.strategy.decide(profile, segments, params, spec) -> list[GenerationJob]`. Segment-wins merge on `Segment.params`. `spec["_audio_mode"]` carries joint-vs-separate audio decision.
- **Validation as a pure function** — `core.validation.validate_request(profile, request, *, accepted_kinds)` returns a NEW `GenerationRequest` via `dataclasses.replace`; never mutates input.
- **TDD red-first, every task** — each task wrote the failing test first, ran it, confirmed FAIL, then implemented. The `test-design` skill convention holds: every test names a concrete bug it would catch in a `# Bug:` comment; assertions verify behaviour, not implementation.
- **Durability** — git is the source of truth; `PROGRESS.md` updated and committed after every task; the `superpowers-extended-cc:subagent-driven-development` skill drove execution (one fresh implementer subagent per task, spec-compliance review, code-quality review, fix loops until clean).
- **`# type: ignore[type-arg]` on bare `dict` annotations** in core types is a deliberate codebase convention (runtime structure is open; parameterising adds no value). `core/interfaces.py` is the precedent; the new `Splitter` ABC + `HeuristicSplitter` follow it.

New patterns introduced by the splitter work:
- **In-core defaults wire themselves up via `core/__init__.py`** — distinct from adapter packages which still import only via `_adapters.py`.
- **`Field(default_factory=NestedModel)`** for optional pydantic nested blocks (`SplitterConfig`), not `Field(default=NestedModel())`. Avoids shared-instance defaults.
- **Asset attachment is an orchestrator concern, not a splitter concern.** Splitter contract stays prompt-only; orchestrator does the seg-0 attach via `dataclasses.replace(prompt_segments[0], assets=list(validated.assets))`. The `list(...)` copy guards against caller aliasing.

## 7. What works today (post-splitter Definition of done)

Every prior MVP Definition of done item still passes. Splitter-specific:
- `cfg.splitter.kind` defaults to `"heuristic"` when the block is absent (4 example configs prove it).
- `splitter:` block parses with both `kind: heuristic` (explicit default) and arbitrary unknown kinds (error deferred to `generate()` via `UnknownAdapter`).

## 11. Process retrospective — Superpowers workflow on the splitter

Discipline that worked well, worth repeating on layer #02:

- **Brainstorm before plan, plan before code.** The validated design contract caught the engine-kind ADR question (parking it on issues #03 + #05) before any code was written.
- **One question at a time** during brainstorming. Multiple-choice with a recommended option keeps tempo without sacrificing clarity.
- **Subagent-driven execution.** One implementer per task + spec-compliance reviewer + code-quality reviewer. The two-stage review caught the `validate_request` ordering issue + the README inconsistency that a single review would have missed.
- **Plan can expand mid-execution.** Task 4 grew to include `pipeline/generate_clip.py` once the once-per-generate AC's interaction with the existing stage code became clear. Better to expand the plan in-flight than to land a half-working integration.
- **NITs get fixed in a polish commit, not deferred.** Code-quality reviewer's NITs landed as `999e39d` (docstring + README example). Cheap, and the audit trail stays clean.

What to watch for next time:

- **PROGRESS SHA pinning interacts badly with `git amend`.** Each amend shifts the SHA, so pinning requires either a follow-up commit or `--autosquash`. Don't keep amending if PROGRESS references the SHA of the commit being amended.
