ABC changes Layer A + Layer B brought:

- **`ArtifactStore.uri_for(run_id, name) -> str`** — new abstract method. Pure, no I/O, deterministic. Equal to the `uri` field of the `Artifact` that `put_bytes` / `put_json` would return for the same `(run_id, name)`. `LocalArtifactStore` delegates to its existing private `_path`. Future S3/GCS impls return `s3://bucket/key` / `gs://bucket/key`. The `hasattr(_path)` peek that lived in `JsonProfileCache._reconstruct_uri` is gone — that whole module's `_uri_index`/`_uri_for`/`_reconstruct_uri` machinery was deleted.

- **`GenerationEngine.extract_last_frame(artifact) -> ConditioningAsset`** — new concrete (NOT `@abstractmethod`) default on the ABC. Raises `NotImplementedError(f"{type(self).__name__} does not support tail-frame extraction")`. FakeEngine overrides with a deterministic synthetic asset. ComfyUI / Diffusers / Hosted inherit the raise — calling continuity on those engines fails fast at chain time. Each of those engines is a follow-up.

## 4. Architectural decisions added this session (locked)

### 4.1 Windows CI scoped-then-declined (commit `03a053e`)

- `pixi.toml` line 32 declares `platforms = ["linux-64", "osx-arm64", "osx-64"]`. Adding `"win-64"` would have required: per-target gating of scaffolding deps that have no win-64 conda-forge build (e.g. `ipython`, `ipdb`, `difftastic` — flagged unavailable by audit), regenerating `pixi.lock`, rewriting one pre-commit hook (`check-added-large-files` uses `bash -c` which has no Windows runner equivalent), fixing one known Windows test blocker (`stores/local.py` `str(Path.relative_to(...))` produces backslashes on Windows but is asserted with `/` in `tests/stores/test_local.py:105`), and auditing the rest of the test suite (378 tests had never been validated on Windows — Windows was silently red since CI was set up).
- The whole plan lives at `windows-migration-cancelled.md` (committed) as audit trail.
- CI matrix is now `[ubuntu-latest, macos-latest]`. The decision is reversible — if Windows support becomes a real goal, the cancelled doc is a starting point.

### 4.2 `ArtifactStore.uri_for` contract (Layer A)

- **Pure, no I/O, no existence check.** Returns the URI a `(run_id, name)` pair *would* address — does not stat / HEAD / list. Callers that care about existence use `list()` or let `get_bytes` raise `FileNotFoundError`.
- **Invariant:** `store.uri_for(run_id, name) == store.put_bytes(run_id, name, b).uri` for any bytes `b`. Same for `put_json`. Tested at the `LocalArtifactStore` level; future S3/GCS impls must satisfy it.
- **`LocalArtifactStore._path` stays private.** `uri_for` is the public API; `_path` continues to exist as the implementation detail that `uri_for`, `put_bytes`, `delete`, etc. all delegate to.
- **`JsonProfileCache` got a full cleanup, not a minimal refactor.** With `uri_for` available as a pure O(1) operation, the `_uri_index` dict, the `_uri_for` helper, the `_reconstruct_uri` method, the `_uri_index` populate inside `_persist`, the class-docstring `_uri_index` entry, and the module-level "URI reconstruction" docstring block all became dead. Net delete ~50 LOC; net add ~3.

### 4.3 Continuity dispatch via `MODE_ROLE_REQUIREMENTS` (Layer B)

- The non-native strategy branch fans out to N single-segment jobs. Layer B inserts tail-frame conditioning between adjacent jobs *only when* `"init_image" in MODE_ROLE_REQUIREMENTS.get(request.mode, set())`.
- Today this evaluates True for `i2v` only. `t2v` (empty contract) and `flf2v` (no `init_image`) both skip chaining. Future modes that adopt `init_image` get chaining for free.
- This decouples continuity logic from the mode-name vocabulary. The role contract is the source of truth.

### 4.4 Continuity loop home: stage owns it; module is a pure helper (Layer B)

- The interleaved render → extract → inject → render loop lives inside `GenerateClipStage.run`. `core/continuity.py` exposes one pure helper: `inject_tail_frame(next_job, prev_artifact, engine) -> GenerationJob`.
- Stage gained an `engine: GenerationEngine` field; the orchestrator's stage construction at `orchestrator.py:477` passes `engine=resolved_engine`.
- The old `pool.map(jobs)` block became a sequential loop: `for i, job in enumerate(jobs): if i > 0 and should_chain: job = inject_tail_frame(...); art = self.pool.submit(job).result(); results.append(art)`. Behaviour for native branch (1 job) and non-chain modes (t2v, flf2v, N=1) is exactly equivalent to the old `pool.map`.
- **Intermediate artifacts are NOT persisted.** Stage continues to persist only `results[-1]`. The N-1 intermediate renders exist in-memory long enough to feed the next iteration's `extract_last_frame`. The stitching layer (separate follow-up) will refactor the persistence contract when it ships.

### 4.5 `extract_last_frame` is a concrete ABC default (Layer B)

- The method is concrete on `GenerationEngine` (NOT `@abstractmethod`) with a default body that raises `NotImplementedError("<EngineName> does not support tail-frame extraction")`.
- Engines opt in by overriding. Only `FakeEngine` does today.
- Multi-segment + non-native + non-overriding engine = `NotImplementedError` at chain time. Acceptable because those engines either support `supports_native_extension=True` (so the chain branch never executes) or get the feature added later in their own follow-up.
- The signature was chosen as `extract_last_frame(self, artifact: Artifact) -> ConditioningAsset` — engine returns the asset ready-to-use with `kind="image"`, `role="init_image"`, `ref=<frame Artifact>`. `inject_tail_frame` does not assemble.

### 4.6 `inject_tail_frame` REPLACES seg-0 assets, doesn't append (Layer B)

- Splitter contract guarantees segs 1..N-1 have `assets == []`. The helper sets `segments[0].assets = [tail_asset]` rather than prepending. If a caller violates the splitter contract (segment already has assets), the behaviour is "replace" — the helper is opinionated.
- Other segments in `next_job.segments` (if any) are passed through unchanged. The helper uses `dataclasses.replace` for both the segment and the job; the input is not mutated.

## 5. Patterns established (use these for future work)

Carries forward from the prior two handoffs (§4 in each); still all in force:
- Injected I/O seams on every adapter (stdlib defaults; spies in tests).
- Self-registration on import.
- Source dispatch by `handles(ref)`, not key equality.
- Stage protocol + pool-swap.
- Strategy as a pure function.
- Validation as a pure function.
- TDD red-first, every task.
- Durability via git + `PROGRESS.md` + atomic commits.
- `# type: ignore[type-arg]` on bare `dict` annotations in core types.
- In-core defaults wire themselves via `core/__init__.py` (added during splitter work).
- `Field(default_factory=NestedModel)` for optional pydantic nested blocks.
- Asset attachment is an orchestrator concern, not a splitter concern.

New patterns introduced by this session:

### 5.1 Brainstorm → spec → plan → execute → ship, fully Superpowers-driven

Both Layer A and Layer B followed the exact same end-to-end flow:

1. `superpowers-extended-cc:brainstorming` skill — one question at a time, multiple-choice with recommendation, locked decisions to a `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` doc committed on `main`.
2. `superpowers-extended-cc:writing-plans` skill — refined to a `docs/superpowers/plans/YYYY-MM-DD-<topic>.md` doc with full step-by-step code blocks, HEREDOC commit messages, exact test bodies. Tasks snapshot (`.tasks.json`) co-located. Committed on `main`.
3. `superpowers-extended-cc:subagent-driven-development` skill — created `build/<topic>` branch off main; per task: dispatch implementer subagent with full task body, two-stage review (spec compliance reviewer + code quality reviewer via `caveman:cavecrew-reviewer`), mark task complete + sync tasks.json, repeat.
4. Final whole-branch review.
5. `superpowers-extended-cc:finishing-a-development-branch` skill — merge `--no-ff` to main, verify post-merge tests, delete build branch.

Both layers shipped in three atomic commits each (feat → feat-or-refactor → docs) plus a chore tasks.json sync commit, matching the splitter's pattern (`089e0a7 chore: mark all 4 splitter tasks completed in tasks.json snapshot`).

### 5.2 Spec self-review tightens vague pointers before commit

While writing the Layer B spec, the section instructing the implementer where to add the engine field to `orchestrator.py` was originally vague ("the orchestrator already holds an engine reference"). Self-review caught it; the spec was tightened in-place to name the line (`orchestrator.py:477`), the local variable (`resolved_engine`), and the exact diff. The reviewer subagents downstream had nothing to flag because the ambiguity was already gone.

Lesson: spec-self-review before committing a spec is cheap and catches "implementer-must-grep" footnotes that would have cost time downstream.

### 5.3 ABC change → grep for construction sites pre-implementation

Adding a required field to a `@dataclass` ABC (`GenerateClipStage` got `engine: GenerationEngine`) breaks every construction site. The plan listed all of them up front (`tests/pipeline/test_generate_clip.py:55,182,202`; `tests/core/test_pool.py:162-163`; `src/kinoforge/core/orchestrator.py:477`) so the implementer had no surprises. Pattern: when adding a required dataclass field, the spec/plan must enumerate every construction site (`rg "ClassName\("`).

### 5.4 In-context container blindness — `git fetch --tags` before claiming a tag is missing

I asserted twice in this session that `v0.2.0` was unTagged after the user had already created and pushed the tag outside the container. Local refs went stale. Saved as a feedback memory at `~/.claude/projects/-workspace/memory/feedback_verify_tags_before_recommending.md`: run `git fetch --tags origin && git ls-remote --tags origin` before claiming a tag or remote-branch fact, especially after the user mentions doing anything outside the container.

### 5.5 Concrete ABC defaults are a legitimate extension pattern (Layer B)

Pre-Layer-B, all 5 abstract methods on `GenerationEngine` were `@abstractmethod`. Layer B added a 6th method (`extract_last_frame`) as a CONCRETE default that raises. The 5 abstracts have semantics every engine must support; the 6th has semantics that some engines can opt into. Mixing concrete and abstract methods on the same ABC is fine when the doc clearly says "subclass to enable; default raises". A future S3/GCS store ABC method that's only meaningful for cloud stores would follow the same pattern.

### 5.6 `dataclasses.replace` for every immutable update

Both layers used `dataclasses.replace` extensively. `inject_tail_frame` builds a new seg-0 via `replace(next_job.segments[0], assets=[tail_asset])` then a new job via `replace(next_job, segments=[new_seg_0, *next_job.segments[1:]])`. No mutation paths exist. Tests verify with `is` identity on unchanged segments and equality checks on the changed one.

### 5.7 Two-stage review catches different classes of issue

Spec compliance reviewer (caveman:cavecrew-reviewer) and code quality reviewer (same subagent type with a different prompt focus) found different things:
- Spec reviewer caught one near-miss on Layer A around the "phantom" `test_cross_restart_read` test (no such test existed pre-refactor; spec was tightened to say "add a cross-instance regression test that fills the missing coverage").
- Quality reviewer focused on placement, imports, mutation paths, test independence — caught nothing in this session, but the rigour matters.
- Order matters: spec first, then quality. A quality-clean commit that doesn't match the spec is still wrong.
