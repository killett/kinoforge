## 4. Architectural decisions added this session (locked)

### 4.1 Two independent sibling adapters, no shared cloud-base class (Approach A from spec §1)

- `src/kinoforge/stores/s3/__init__.py` and `src/kinoforge/stores/gcs/__init__.py` are siblings of `src/kinoforge/stores/local.py`. There is NO `CloudArtifactStore` middle abstract class.
- Each cloud store implements the 7-method ABC directly. Approach B (shared cloud base) was explicitly rejected during brainstorm; the small (~30 LOC) duplication across the two files is acceptable, and the middle layer would have locked in guesses about future stores (Azure? B2?). If a third cloud store ever lands, that's the time to factor.
- Pattern matches every other adapter pair in repo (RunPod pod vs serverless, ComfyUI vs Diffusers, etc).

### 4.2 Sync SDKs, not async (boto3 + google-cloud-storage)

- The `ArtifactStore` ABC is sync. Switching to async would have required a parallel async ABC + downstream rewrite of `JsonProfileCache`, `GenerateClipStage`, `Ledger`. Blast radius too high for a layer that should plug into existing seams.
- `aiobotocore` + `gcloud-aio-storage` deferred indefinitely. Real driver for async would be high-fanout uploads (e.g. concurrent batch generation) — Layer #4 (ConcurrentPool, issue #3) might raise the question, but a sync store behind a threaded ConcurrentPool likely covers it.

### 4.3 SDK default credential chain only, no kinoforge cred plumbing

- `S3ArtifactStore(client=None)` calls `boto3.client("s3")` which walks: env (`AWS_ACCESS_KEY_ID` etc), `~/.aws/credentials`, EC2 IAM role, ECS task role, etc.
- `GCSArtifactStore(client=None)` calls `google.cloud.storage.Client()` which walks: `GOOGLE_APPLICATION_CREDENTIALS`, gcloud ADC, GCE metadata, etc.
- Routing through `EnvCredentialProvider` was explicitly rejected — it would defeat IMDS/IAM-role auto-discovery, breaking users on EC2/GCE compute. Cred provenance is opaque to kinoforge; cloud-correct behaviour out of the box.
- Future requests for explicit per-region clients or session tokens get a follow-up issue; the layer ships with the most common path correct.

### 4.4 SDK-default multipart auto-switch, no kinoforge knob

- `put_bytes` on S3 uses `put_object(Body=data)` (boto3 auto-switches to multipart over its ~8 MiB threshold for `upload_fileobj`; `put_object` keeps single-PUT semantics). For GCS, `blob.upload_from_string(data)` auto-multiparts over the SDK threshold.
- No `multipart_threshold_mb` config knob. Layer C explicitly ships for the profile-JSON + small-clip workload; GB-scale uploads work because the SDKs handle it, but the kinoforge surface stays flat.

### 4.5 Injected client seam + spy fakes (test invariant)

- Each cloud store accepts a `client` kwarg defaulting to `None` (real SDK lazy-imported on first construction); tests pass a `FakeS3Client` / `FakeGCSClient` (in-memory dict-backed).
- `FakeS3Client` exposes: `put_object`, `get_object`, `head_object`, `delete_object`, `get_paginator("list_objects_v2")`, `.exceptions.NoSuchKey`, `.exceptions.ClientError(code)`. Stream-body double exposes `read() -> bytes`.
- `FakeGCSClient` exposes: `bucket(name) -> _FakeBucket` which has `blob(key) -> _FakeBlob` and `list_blobs(prefix=...)`. `_FakeBlob` has `upload_from_string`, `download_as_bytes`, `delete`, `exists`. `FakeGCSClient.NotFound` is the exception class.
- Both fakes live in a single shared `tests/stores/conftest.py` — landed in Task 1 alongside `FakeS3Client` even though `FakeGCSClient` isn't consumed until Task 2. Single conftest avoided two edits.
- Lazy-import gate proven by `sys.modules` snapshot tests (1 per store). For GCS, the snapshot covers BOTH `google.cloud.storage` AND `google.api_core.exceptions` because the `NotFound` exception class is the second lazy gate.

### 4.6 Dual lazy-import gate for GCS (`client` + `not_found_exc`)

- `GCSArtifactStore.__init__(client=None, not_found_exc=None)`. When `client is None`, lazy-imports `google.cloud.storage`. When `not_found_exc is None`, lazy-imports `google.api_core.exceptions.NotFound`. Tests must inject BOTH to satisfy the offline-test invariant.
- S3 has only one gate because the `NoSuchKey` / `ClientError` classes are exposed via `client.exceptions.*` (boto3 attaches them to the client object). GCS exception hierarchy lives in a separate module, so the second seam was necessary.
- This is the spec self-review catch (commit `e95b94d`) — the original spec assumed a single gate; the dual-gate pattern was clarified before the plan was written, so the plan and implementation both got it right first time.

### 4.7 Error mapping: misses → `FileNotFoundError(uri)`, rest propagates

- S3 `get_bytes` catches `client.exceptions.NoSuchKey` → `FileNotFoundError(uri)`. `delete` does a `head_object` pre-check (since S3 `delete_object` is idempotent by design), translates ClientError codes `404|NoSuchKey|NotFound` → `FileNotFoundError(uri)`.
- GCS `get_bytes` and `delete` catch the injected `NotFound` exception class → `FileNotFoundError(uri)`. No pre-check needed because GCS `blob.delete()` raises NotFound on missing key.
- Auth, throttling, network, all other SDK errors propagate as-is. Caller can catch SDK exceptions for SDK-specific handling. Honours the `ArtifactStore` ABC contract minimally.
- Retry / backoff knobs explicitly deferred — both SDKs already retry transient errors. Adding a kinoforge retry layer would cause behavioural divergence from `LocalArtifactStore` for no clear gain.

### 4.8 Key layout: `<scheme>://<bucket>/<prefix>/<run_id>/<name>`

- Mirrors `LocalArtifactStore`'s `<root>/<run_id>/<name>` exactly. `prefix` is normalised to have no leading/trailing slash; empty prefix produces no leading slash on the key.
- `list(run_id)` builds `run_prefix = self._key(run_id, "") + "/"` and uses S3 paginator / GCS `list_blobs(prefix=...)`. The trailing slash is load-bearing: it prevents `run-1` from accidentally including items under `run-10/`. Test `test_list_excludes_other_run_ids` pins this.

### 4.9 Optional `StoreConfig` pydantic block; absent block = LocalArtifactStore(state_dir)

- New field `Config.store: StoreConfig = Field(default_factory=StoreConfig)`. `StoreConfig` has `kind: Literal["local","s3","gcs"]` (default `"local"`), `root: Path | None`, `bucket: str | None`, `prefix: str` (default `""`).
- `model_validator(mode="after")` enforces: `kind in {"s3","gcs"}` requires `bucket`; `kind == "local"` rejects `bucket`. Both directions guarded so a typo doesn't slip through.
- Configs without a `store:` block parse identically to pre-Layer-C — backwards compatible. Only `examples/configs/wan.yaml` gained a commented `store:` block (4 sub-sections: header + explicit-local + s3 + gcs); other example configs omit it on purpose to prove the default path still works.
- `kinoforge.gc` subcommand gained required `--config PATH` arg as a consequence (it didn't previously load a `cfg`; needed one to route through `_build_store`). Documented in PROGRESS.md Phase 13.

### 4.10 CLI `_build_store(cfg, state_dir)` dispatcher with lazy SDK imports per branch

- Single helper in `cli.py` (placed next to `_ledger`). Dispatches by `cfg.store.kind`:
  - `local` → `LocalArtifactStore(sc.root or state_dir)` (the `or state_dir` keeps `--state-dir` as the fallback root)
  - `s3` → `from kinoforge.stores.s3 import S3ArtifactStore; S3ArtifactStore(bucket=sc.bucket, prefix=sc.prefix)` (lazy import inside the branch — keeps CLI startup fast on non-S3 paths)
  - `gcs` → analogous
  - else → `raise UnknownAdapter(...)` — pydantic Literal should have caught it earlier, but the explicit raise covers `Config.model_construct(...)` paths in tests
- `_cmd_generate` (CLI command body) and `_cmd_gc` both go through `_build_store`. The two `LocalArtifactStore(state_dir)` direct constructions previously at `cli.py:265` and `cli.py:434` are gone.
- **`_ledger(state_dir)` is byte-for-byte unchanged.** Ledger is *operator* state (which compute instances exist, which are reapable) — distinct from *run* output and orthogonal to the store-selection axis. Cloud-backed ledger intersects issue #7 (cross-process discovery lock) and stays out of scope.

### 4.11 Layer-A leftover `cli.py:441 store._path` peek finally retired

- `_cmd_gc`'s inner loop at the old line 441 used to call `str(store._path(run_id, name))` — a private-attr peek into `LocalArtifactStore` that violated the same encapsulation Layer A's `uri_for` ABC was designed to fix. Layer A retired the equivalent peek in `JsonProfileCache._reconstruct_uri` but missed this site.
- Layer C swaps it for `store.uri_for(run_id, name)` — public ABC method, no `str()` wrapper needed (the ABC contract already returns `str`).
- Pinned by `test_cli_gc_uses_store_uri_for_not_path_peek` which greps `cli.py` source to forbid the regression. `grep "store\._path" src/kinoforge/cli.py` returns 0 matches.

### 4.12 `assert` banned in `src/` (ruff S101)

- Plan author wrote `assert sc.bucket is not None` for type-narrowing inside `_build_store`. Ruff S101 flagged it; implementer replaced with explicit `if sc.bucket is None: raise ValueError(...)`.
- This is a project-wide rule (S101 is enabled in `pyproject.toml`); it's also stricter (the explicit raise persists under `python -O`). Documented as a style precedent for future cloud-store work.

### 4.13 `boto3` lacks `py.typed`; `google.api_core` ships it

- `S3ArtifactStore`'s lazy `import boto3` needs `# type: ignore[import-untyped]` to pass mypy strict. `GCSArtifactStore`'s lazy `import google.cloud.storage` needs the same. But `google.api_core.exceptions.NotFound` does NOT need a type-ignore because `google-api-core` ships `py.typed`.
- Asymmetry caught during Task 2 implementation; documented inline. If future cloud SDKs are added, check `py.typed` before pre-emptively adding type-ignore.

### 4.14 `import X.Y.Z as alias` form preferred over `from X.Y import Z` for lazy SDK imports

- Task 2 implementer found that ruff-format wraps `from google.cloud import storage  # type: ignore[import-untyped]  # noqa: PLC0415` onto multiple lines, separating the type-ignore comment from the line mypy reports the error on.
- Workaround: `import google.cloud.storage as _gcs_storage  # type: ignore[import-untyped]  # noqa: PLC0415` keeps the comment attached.
- Lazy-gate test still passes because `import google.cloud.storage as alias` populates `sys.modules["google.cloud.storage"]` (the exact module name the test snapshots). Vendor invariant regex `r"^\s*(import|from)\s+google\.cloud\b"` also matches.
- Documented style precedent for any future cloud SDK lazy import.

## 5. Patterns established (carry-forward + new from this session)

Carrying forward from prior handoffs (§4 in each); still all in force:
- Injected I/O seams on every adapter (stdlib defaults; spies in tests).
- Self-registration on import.
- Source dispatch by `handles(ref)`, not key equality.
- Stage protocol + pool-swap.
- Strategy + validation + continuity helpers as pure functions.
- TDD red-first, every task.
- Durability via git + `PROGRESS.md` + atomic commits.
- `# type: ignore[type-arg]` on bare `dict` annotations in core types.
- In-core defaults wire themselves via `core/__init__.py` (splitter pattern).
- `Field(default_factory=NestedModel)` for optional pydantic nested blocks.
- Asset attachment is an orchestrator concern, not a splitter concern.
- Brainstorm → spec → plan → execute → ship discipline (Layer A/B).
- Spec self-review tightens vague pointers before commit (Layer A/B).
- ABC change → grep for construction sites pre-implementation (Layer B).
- `git fetch --tags` before claiming a tag is missing (memory).
- Concrete ABC defaults are a legitimate extension pattern.
- `dataclasses.replace` for every immutable update.
- Two-stage review per task: spec compliance first, then code quality.
- `--no-ff` merge pattern with substantive merge-commit body referencing `Closes #N`.

New patterns introduced by Layer C:

### 5.1 Cloud SDK lazy-import gate (single + dual variants)

- Constructor `__init__` takes the SDK client as an optional kwarg defaulting to `None`. Inside `if client is None:` block, `import <sdk>` happens (with `# noqa: PLC0415 — lazy` comment for ruff).
- For SDKs where the exception class lives in a different module (e.g. GCS), a second seam `not_found_exc=` covers it. BOTH gates fire under their `is None` checks. Tests inject both to bypass.
- Lazy-gate proven by `sys.modules.pop(sdk_name, None); construct(client=fake); assert sdk_name not in sys.modules`. One test per gated module.

### 5.2 Shared test conftest for sibling adapters

- `tests/stores/conftest.py` holds BOTH `FakeS3Client` and `FakeGCSClient` even though Task 1 only uses S3 and Task 2 only uses GCS. Avoids two conftest edits across sibling-adapter tasks.
- Each fake is minimal — only the SDK surface the adapter actually calls. No simulation of pagination beyond a single page, no SSE/KMS, no signed URLs.

### 5.3 Optional config block with backwards-compat default via `Field(default_factory=NestedModel)`

- `Config.store: StoreConfig = Field(default_factory=StoreConfig)`. When YAML lacks `store:`, the field constructs with all-default `StoreConfig` (kind=local, root=None, bucket=None, prefix=""). Existing configs parse identically to pre-Layer-C — load_config never raises on absent block.
- Tests cover both directions: absent block → default; explicit block round-trips.
- Same pattern was used for the splitter's `SplitterConfig` (Phase 10); now formally established.

### 5.4 `model_validator(mode="after")` for cross-field invariants

- StoreConfig's `_check_kind_requirements` enforces `kind in {"s3","gcs"} → bucket required` AND `kind == "local" → bucket rejected`. Both directions in one validator.
- Tests exercise the validator with `pytest.raises(ValidationError, match="...")` — match string targets the specific error message so a different validator firing won't false-positive.

### 5.5 Test annotation discipline (real types, not `# noqa: ANN001` placeholders)

- Even with `[[tool.mypy.overrides]] module = ["tests.*"] strict = false`, mypy still flags `no-untyped-def` on test functions without parameter annotations. The override does NOT suppress that check.
- Plan authoring style: use `# noqa: ANN001` placeholders for brevity. Implementer style: switch to real annotations (`def test_x(store: S3ArtifactStore) -> None:`) matching the prevailing `tests/stores/test_local.py` shape.
- Layer C reviewers approved this deviation on Tasks 1, 2, 3, 4 — formalised as a precedent.

### 5.6 CLI dispatcher with lazy SDK imports per branch

- Pattern: a top-level `_build_X(cfg, ...)` helper inside `cli.py` that dispatches by a `Literal` field. Each cloud-SDK branch does the SDK import lazily inside the branch body. Non-cloud branches stay import-free.
- Keeps CLI startup fast: a user running `kinoforge generate` against a `kind: local` config never imports boto3 or google.cloud.storage.

### 5.7 Quality reviewer escalating NICE → FIX-REQUIRED is fair game

- Layer C Task 4 quality reviewer flagged a docstring `Raises:` omission as FIX-REQUIRED (not NICE) because the unhandled-case branch is reachable via `Config.model_construct()` in tests. Implementer (via a small fix subagent) added the missing exception entry; reviewer approved on second pass.
- Pattern: if a documented `Raises:` is incomplete AND the unraised path is reachable by ANY test or supported caller pattern, it's a FIX-REQUIRED gap, not a polish nit.

### 5.8 Builder subagent without Bash → controller handles verify-and-commit

- `caveman:cavecrew-builder` ships without Bash access (read-only edit surface). For a 2-line docstring fix, dispatching it for the edit is fine; the controller then runs pre-commit + tests + commits itself.
- Pattern: use the builder for surgical edits; never expect it to run verification or commit.

### 5.9 Plan headline counts can drift from actual test counts

- Layer C plan said "17 tests" per cloud store; the verbatim code blocks in the plan actually had 18 tests each (the `uri_for` invariant was split into `put_bytes` + `put_json` variants — spec §8.2 item 15 explicitly says "and put_json variant"). Implementers caught and reported; no fix needed.
- Lesson: plan-section titles are summary-grade; the verbatim code blocks are the contract. When they diverge, the code blocks win.

### 5.10 Helper signature changes that emerge during implementation (`kinoforge gc --config`)

- Task 4 plan Step 5 instructed: "if `_cmd_gc` lacks `--config` arg or cfg loading, add the pattern from `_cmd_generate`". This contingency was anticipated up-front — and exercised. `gc` became `gc --config PATH`-required. Documented in commit message + PROGRESS.
- Pattern: when adding a feature that depends on a `cfg` to a command that didn't previously load one, plan for the signature change in the plan, not as a surprise during implementation.

### 6.3 The one real fix loop (Task 4 docstring)

## 8. Known limitations (post-Layer-C)

Carrying forward + revising the prior list:

1. ~~`JsonProfileCache._reconstruct_uri` peeks at `LocalArtifactStore._path`~~ — **resolved by Layer A.**
2. `cli._cmd_status` queries only in-process provider state, not the ledger.
3. `provisioner.provision` takes `cfg: Config` typed as `_ProvisionConfig` Protocol; `# type: ignore[arg-type]` at the call site for mypy generic variance.
4. `RunPodProvider.find_offers` REST shape is a stub — real-cloud usage will need a small JSON adapter once pointed at a live key.
5. SkyPilot provider ships the structure + injected client seam; lazy `_get_sky()` real path is wired but unexercised against the real `sky` SDK.
6. `test_core_invariant.py` allowlist does not yet include a `splitters/` directory. First time an adapter splitter is added (LLM-semantic, scene-detect), extend the allowlist.
7. `ComfyUIEngine`, `DiffusersEngine`, `HostedAPIEngine` inherit the `extract_last_frame` default that raises. Non-native multi-segment + any of those = `NotImplementedError` at chain time. Each engine's `extract_last_frame` impl is a follow-up.
8. `GenerateClipStage` returns only `results[-1].uri` from the store; intermediate segment artifacts are in-memory only. Stitching, when it ships, will need to refactor either the stage's persistence model or the stitching layer's read path.
9. `flf2v + N > 1 + non-native` is a pre-existing gap.
10. CI Windows declined; rationale in `windows-migration-cancelled.md`.

New post-Layer-C:

11. **`_cmd_gc` is now `--config`-required.** Pre-Layer-C, `kinoforge gc --run <rid>` worked without a config file. Now `--config PATH` is mandatory because gc routes through `_build_store(cfg, state_dir)`. Anyone with shell scripts calling `kinoforge gc` without `--config` will need to update them. Documented in PROGRESS Phase 13. The breaking-change tradeoff was accepted at planning time — alternative (defaulting cfg to a builtin local-only stub) would have been an extra code path with unclear semantics.
12. **Ledger remains local-only.** `cli._ledger(state_dir)` is byte-for-byte unchanged; it always constructs a `LocalArtifactStore(state_dir)` for the `ledger.json` blob. A user with `store.kind: s3` still has their compute-lifecycle ledger on local disk. Cloud-backed ledger intersects issue #7 (cross-process discovery lock) and is explicitly deferred. For single-node users this is invisible; for multi-node distributed setups it's a hard wall until both layers ship.
13. **`S3ArtifactStore` / `GCSArtifactStore` are unverified against real cloud.** The whole suite is offline. Real-cloud smoke test (one bucket, one round-trip, one delete) is a follow-up item that should run before the first production user hits the path. The fake clients are minimal — anything boto3/google-cloud-storage does that the fakes don't simulate (multipart chunking edge cases, transient retries, SSE) is uncovered.
14. **Default zero-arg factories require env vars set.** `register_store("s3", _default_factory)` reads `KINOFORGE_S3_BUCKET` from env; `_default_factory()` (called by `get_store("s3")()`) raises `RuntimeError` when unset with a helpful message pointing at direct construction. The CLI doesn't use the factory path — it constructs `S3ArtifactStore(bucket=cfg.store.bucket, prefix=cfg.store.prefix)` directly via `_build_store`. The factory path exists for library users wanting "just give me a store named s3". Worth documenting in README if users get confused.
15. **No multipart threshold knob.** SDK defaults (boto3 ~8 MiB, GCS similar). For workloads with non-default upload patterns (very small chunks, custom SSE, large parallel writes), users wanting control need to construct the store directly and pass extra `TransferConfig` / `chunk_size` via kwargs that don't yet exist. Layer C ships with the most common path correct; knobs come later if real workloads need them.

## 11. Process retrospective — Superpowers workflow on Layer C

Discipline that worked well, worth repeating on future layers:

- **Spec self-review caught two ambiguities** before the plan was written: (a) the CLI test file path (`tests/cli/test_cli.py` doesn't exist; correct is `tests/test_cli.py`) — wrong path would have wasted implementer time; (b) GCS dual-gate semantics — the original spec had a single `not_found_exc` ambiguity that the self-review tightened to "BOTH client AND not_found_exc must be injected together". Spec commit `1f8016d`, self-review fixes `e95b94d`. The combined cost was ~10 minutes of self-review; the saved cost was at least one round-trip per task.
- **Plan included the contingency for `_cmd_gc` --config.** Task 4 plan Step 5 explicitly said: "if `_cmd_gc` lacks `--config`, add the pattern from `_cmd_generate`". Implementer hit exactly that case and resolved it without escalating because the plan already authorized the change.
- **Two-stage review (spec then quality) caught the Task 4 docstring gap** that neither implementer self-review nor the spec reviewer flagged. Quality reviewer noticed: `_build_store` raises `ValueError` for `bucket is None` on the s3/gcs branches, but the docstring only listed `UnknownAdapter`. Reachable via `Config.model_construct()` in the tests. Escalated NICE → FIX-REQUIRED because the unraised path is reachable by a supported test pattern. One commit (`b661576`) closed the loop.
- **`--no-ff` merge commit body summarises the layer.** `c0b401e`'s 30-line merge body covers what the layer does, all internal commit SHAs, the breaking change for `gc`, the test/coverage/lint state, and the `Closes #5` invariant. Future PROGRESS / handoff references point at the merge commit as the layer boundary.
- **Per-task `Refs #N` + merge-commit `Closes #N`** — every per-task feature commit carries `Refs #5`; only the merge commit carries `Closes #5`. Matches Layer A + Layer B pattern. The trailer fires once when pushed.

What to watch for next time:

- **Container `gh` is unavailable; user pushes/tags/comments outside.** Local refs go stale between turns. Saved feedback memory at `~/.claude/projects/-workspace/memory/feedback_verify_tags_before_recommending.md`.
